Events & Listeners

MCIdentity has two sides to its listener system: the custom Bukkit events your own plugin can hook into, and the internal gameplay listeners that translate player actions into profession experience.

How listeners work

The plugin provides several custom events in the io.github.mcoriax.mcidentity.bukkit.event package. Every one of them supports Folia's asynchronous context and works across all supported platforms — SpigotMC, PaperMC, and Folia. An event marks itself asynchronous whenever it is fired off the primary thread, so third-party handlers must stay thread-safe and schedule back onto the appropriate scheduler before touching main-thread-only Bukkit API.

To react to an event, register a normal Bukkit Listener from your own plugin and annotate a handler method — exactly as shown on the Developer API page. The reference below documents each event's purpose, the data it carries, and whether it can be cancelled.

Custom Bukkit events

Every event extends a common identity-event base and is fired for third-party plugins to observe. Expand each one for its details.

MCIdentityEvent base

The abstract base type for all identity events. It automatically handles asynchronous state based on whether it is fired on the primary thread, so every event below behaves correctly on Folia as well as Spigot and Paper.

Internal gameplay listeners

Alongside the public events, MCIdentity runs a set of modular gameplay listeners that award profession experience for what players do. They are grouped by category so each stays focused on one kind of action, and every reward respects anti-farm safeguards (location-once rewards, rate limiting, and vanilla gating). The exact EXP values are defined per profession in YAML — see Professions & EXP.

EXP-gain listeners

Modular listeners grouped by category (combat, defense, gathering, crafting, exploration, farming, and more) translate player actions into profession EXP according to each profession's configuration.

Identity, inventory & stats listeners

Handle equipping and unequipping identities, isolating each identity's inventory and ender chest, and applying stat-driven behaviour.

Respawn stat restoration

Restores an identity's stats after respawn, keeping health, energy, and food consistent with the active identity.

Cold-boot login protection

On the async pre-login thread, pings the central server (when database.type is central) and politely turns players away while a cold-started server is still waking up or storage is still initializing.
