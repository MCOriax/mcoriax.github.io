Data Storage

The plugin supports multiple storage backends, selected with database.type in config.yml: SQLite (the default), MySQL, PostgreSQL, MongoDB, and the MCIdentity central server.

Standalone servers can run entirely on the zero-configuration SQLite file, while networks can share one MySQL, PostgreSQL, or MongoDB deployment — or point every node at the central REST server.

Choosing a backend

SQLite sqlite

Local file in the plugin data folder. Best for single servers; works out of the box with no setup.

MySQL mysql

MySQL or MariaDB server. Best for networks that already run a MySQL-compatible database.

PostgreSQL postgresql

PostgreSQL server. Best for networks that prefer PostgreSQL.

MongoDB mongodb

MongoDB deployment. Best for document-oriented storage or existing MongoDB infrastructure.

Central server central

MCIdentity central REST server. Best for whole networks sharing one authenticated API with token roles.

Each backend has its own connection section in config.yml:

database:
  type: sqlite
  sqlite:
    file: "mcidentity.db"
  mysql:
    host: "localhost"
    port: 3306
    database: "mcidentity"
    username: "root"
    password: "password"
  postgresql:
    host: "localhost"
    port: 5432
    database: "mcidentity"
    username: "postgres"
    password: "password"
  mongodb:
    uri: "mongodb://localhost:27017"
    database: "mcidentity"

The central type reads the existing central_server section (url, token, timeout_seconds).

How it works

  • All backends implement the same IMCIdentityDB contract, so gameplay code never knows which storage is active.
  • The SQLite, MySQL, and PostgreSQL backends share their query logic and differ only in schema DDL and upsert dialect. MongoDB stores each identity as one document with embedded properties, stats, cosmetics, and equipped slots.
  • The central-server backend implements the same contract by sending asynchronous JSON requests over a single pooled HTTP client, carrying the bearer token configured in config.yml (central_server.token).
  • Every operation runs asynchronously on a dedicated worker pool, so the game thread is never blocked.
  • Reads that fail resolve to safe fallbacks (empty results or zero) so the game server keeps running; failures are logged.
  • Stat identifiers are validated against a fixed allow-list before they are used, on the plugin and (for the central server) on the API as well.

Drivers

The SQLite and MySQL JDBC drivers ship with SpigotMC, PaperMC, and Folia. The PostgreSQL driver and the MongoDB client are packaged inside the plugin JAR, so no backend requires extra downloads.

Schema

The relational backends create their tables automatically on first start:

identity_professions identity_abilities identity_runes identity_keys identity_properties identity_stats identity_equipped_abilities identity_equipped_runes identity_sessions identity_equipped_cosmetics identity_logs player_base_slots

The SQLite schema matches the central server's, so a database file can be moved between a game server and a central server deployment.

Data stored

ResourceContents
Professions, abilities, runesThe shared registries identities reference.
Identity keysEach identity's UUID and profession.
PropertiesLevel, experience, stat points, textures, and serialized inventory / ender chest storage.
StatsThe nine numerical attributes (energy, health, food, strength, defense, wisdom, luck, agility, resilience).
Equipped abilities and runesSlot assignments per identity.
CosmeticsActive skin (Base64) and aura (system code) per identity.
SessionsWhich identity each player currently wears, with the serialized identity item.
Base identity slotsHow many alternative base "player" identities each player has unlocked.
LogsThe append-only audit trail (EQUIP, UNEQUIP, DEATH_DROP, ADMIN_REMOVE).

See the central server repository for the REST API reference, environment configuration, and its storage details.
