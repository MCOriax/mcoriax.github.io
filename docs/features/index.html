Key Features

A high-level tour of what MCIdentity brings to your server.

🧱 Multi-Platform Support

Fully supports SpigotMC, PaperMC, and Folia, including native compatibility with Folia's regionalized multi-threading architecture.

🗂️ Interactive GUI System

Multi-tier virtual menus for managing identities, runes, and abilities. See the GUI System page for the full layout.

⚔️ Native MythicMobs Integration

Abilities map directly onto MythicMobs — each ability stores an ability code (the MythicMobs ability/skill ID) and an ability skin (the model ID, with a default-skin fallback). Equipping an ability records exactly which MythicMobs ability runs and which skin renders.

🔄 Dynamic Data Binding

Ability and Rune GUIs fetch real-time data from the database, showing equipped items or placeholders.

🧵 Regionalized Threading

Fully compatible with Folia's RegionScheduler and GlobalRegionScheduler.

🗄️ Multi-Database Storage

Identity data can live in SQLite (the zero-configuration default), MySQL, PostgreSQL, or MongoDB, selected with database.type in config.yml. All backends run fully asynchronously behind the same contract. See Data Storage.

🌐 Central Server Storage

Alternatively, all identity data can live on the MCIdentity central server, accessed through an authenticated REST API over a pooled HTTP client — so every game server on the network shares one consistent data set.

🧩 Extensible API

Provides a central MCIdentityProvider for third-party plugins to interact with the identity system. See the Developer API.

🔐 Secure Commands

Role-based permissions for administrative actions.

🪪 Visual Identity Keys

Identities are represented as physical items in the game with custom colored display names.

🔓 Unbound Identities

Identity items created via /identity get are initially "unbound" — they lack a unique ID and database entry. Both the unique ID and the identity's initial stats are generated and registered in the database only when the item is first used.

👤 Base "Player" Profession

Every player permanently owns a built-in player profession, created automatically on first join and keyed by the player's own UUID. It is the fallback state whenever an identity is unequipped, dropped on death (only when drop_on_death is enabled — false by default), or fails to equip, so a player is never without a profession. It can never be deleted or turned into an identity key item.

Note: the base profession is strictly hardcoded in memory. It does not generate a professions/player.yml file and cannot be modified via YAML.

🎭 Alternative Base Identities

On top of the Primary base identity — the one player profession, which keeps the player's own UUID — players can unlock additional, fully independent identities of any other profession (random or chosen). They are indexed 1…n and keyed by deterministic UUIDs derived from the player UUID and index. Admins grant slots with /identity base add; all owned base identities appear in the paginated Player Identity GUI and can be equipped by clicking their heads.

❄️ Cold-Boot Login Protection

When database.type is central, the plugin pings the central server's /health endpoint during the async pre-login phase — the ping doubles as the wake-up signal for a cold-started server. Players are politely refused while storage initialization has not finished, or while the server is offline, so nobody joins with unreachable data. Initialization retries automatically every 10 seconds. All checks run off the main game thread.

📈 Passive Base Leveling

Base identities level without limit from passive playtime EXP on two tracks. The Primary identity gains 1 EXP per second for every online player — even while another identity is equipped — and levels at 86,400 EXP per level (one level per day of play). An Alternative identity gains its 1 EXP per second only while equipped, and levels at 3,600 EXP per level (one level per hour of active play). Gains are buffered in memory and flushed in bulk every five minutes and on quit.

🚫 Base Profession Restrictions

The base profession may equip skins and auras but is blocked from using runes and abilities.

🎒 Always-On Storage Isolation

Each identity's inventory and ender chest are always isolated and stored with the identity; the previous features.inventory / features.enderchest config toggles have been removed.
