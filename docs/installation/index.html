Getting started

Installation & Setup

The plugin runs on SpigotMC, PaperMC, and Folia servers. Use the build for your platform and follow the steps below.

Steps

  1. Place the plugin JAR in your server's plugins/ folder.

  2. Start the server once to generate the default configuration.

  3. Pick a storage backend in config.yml (see Data Storage). The default is SQLite, which needs no setup at all:

    database:
  type: sqlite

    MySQL, PostgreSQL, and MongoDB are configured in their sections under database. To share the MCIdentity central server across a network instead, set database.type: central and point the plugin at your deployment:

    database:
  type: central
central_server:
  url: "https://server.com"
  token: "the value of SERVER_API_TOKEN on the central server"
  timeout_seconds: 10

  4. Restart the server. The plugin initializes the selected storage on startup and logs an error if it is unreachable.

Death behaviour

By default a player keeps their equipped identity when they die and continues wearing it after respawning. To make the identity key drop on death instead — sending the player back to the base player profession — set:

drop_on_death: true

drop_on_death is false by default.

🌐

With a shared backend (MySQL, PostgreSQL, MongoDB, or the central server), every game server on the network uses the same connection settings, so identities follow players between servers automatically.

Requirements

  • Java 25 or newer (the Gradle build provisions its own JDK 25 toolchain).
  • A SpigotMC, PaperMC, or Folia server (build 26.1.x API line).
  • This project is a plugin extension for those servers — it is not a standalone application.
  • No external database is required by default (SQLite). MySQL, PostgreSQL, MongoDB, or a reachable MCIdentity central server are needed only when configured as the backend.

Testing

The plugin includes a suite of unit tests for the common module to ensure data integrity and logic correctness. To run them, use the following Gradle command:

./gradlew :common:test

The tests cover:

  • Core managers (MCIdentityManager, MCIdentityProvider).
  • The central server client (MCIdentityCentralServer), exercised against a stub HTTP server.
  • Data serialization and session-management logic.
